    记一次http缓存bug

    vuePress-theme-reco 一路败北    2021 - 2023

    记一次http缓存bug

    一路败北 2/5/2023 bug

    # 1. 场景

    html 里面有两个标签

    <link rel="stylesheet" href="http://192.168.153.250:3001/css/index.css" />
<script src="./js/index.js"></script>
    1
    2

    js 文件的内容，axios 发起一个请求加载 css 文件，这个文件和上面 link 加载的是同一个

    axios.get('http://192.168.153.250:3001/css/index.css')
    1

    实际场景模拟的使用了css-vars-ponyfill库的效果，这个库要拿到页面的 css 文件所以得使用 js 发起一次请求

    # 2. 问题

    这个 css 默认是强缓存 1 年，实测生效，多了一个 js 的请求后，缓存失效了，并且这个 js 的请求也不会命中缓存，如图所示，缺少了 from disk

    期望是首次，link 请求没有缓存，xhr 请求有缓存，非首次，两个请求都命中缓存，如图所示，首次加载红框资源应该有缓存

    # 3.排查过程

    • 检查响应头 vary 字段，为什么检查 vary 看这里[RFC 9110: HTTP Semantics (rfc-editor.org) (opens new window)]
      vary: Origin
      1
    • link 请求头没有带 origin，因为 link 请求不会跨域，浏览器默认不携带，但是 xhr 请求头带了 origin，那说明 xhr 请求无法命中 link 请求的缓存？
    • 如果 xhr 请求无法命中 link 请求的缓存，那么在第一次请求的时候，xhr 没命中前一次 link 请求缓存正常，但是第二次请求 xhr 应该命中自己上一次的缓存。
    • 之所以有上一点的猜测，是因为我认为缓存的 key 应该是资源 url+vary 要检查的内容，这种场景就是资源 url+请求头的 origin 字段内容作为 key，但不是这样
    • 那么缓存的 key 应该是什么？看这里[RFC 9111: HTTP Caching --- RFC 9111：HTTP 缓存 (rfc-editor.org) (opens new window)]
    • 从上一点可以总结出来，就是一个资源 URI 对应一个缓存内容（规范是这么写，但 Chrome 不是这么实现），然后会有各种规则开始校验这个缓存内容
    • 所以结论就是 link 请求的缓存，被 xhr 请求命中了，但是 vary 校验不通过，那么 xhr 这次请求会覆盖缓存，下一次 link 请求，命中 xhr 的请求，vary 校验不通过。

    # 4.解决方式

    • xhr 请求后面拼接 query 参数，让两个请求的 URI 不同
    • link 标签加 crossorigin 属性（选了这种），因为 xhr 请求能成功，肯定没有了跨域问题
