    webpack热更新

    一路败北 11/6/2022 webpack

    1. webpack-dev-server 的作用

      • 在 webpack 打包入口注入热更新相关文件，一是 websocket 初始化相关逻辑，二是处理热更新的相关逻辑。
      • 注入 HotModuleReplacementPlugin 插件
      • 初始化各种中间件，比如 webpack-dev-middleware、静态资源中间件
      • 初始化 http 服务、初始化 websocket 服务

    2. webpack-dev-middleware 的作用

      • 以监听模式启动 webpack，并且把打包的结果写到内存中，当有请求进来的时候，从内存中返回内容

    3. webpack-hot-middleware 容易混淆的中间件又是什么

      • 当我们不使用 webpack-dev-server 启动，比如定制化服务端逻辑的场景，就可以使用这个中间件帮我们实现热更新的功能。这个中间件和 webpack-dev-server 功能类似，在 webpack 编译完后，通知客户端处理。但是使用的不是 websocket，而是 SSE。客户端入口文件、HotModuleReplacementPlugin 插件也需要我们手动配置。

    4. 客户端的逻辑

      • 和 vite 相反，webpack 热更新客户端逻辑会更重。
      • 客户端 websocket 收到更新通知，触发 webpackHotUpdate 事件，交给 webpack 处理
      • 通过上次打包的 hash，拼接 url 请求本次要更新的信息，主要有 chunkId
      http://127.0.0.1:3000/app.eb5adff19efc4704908a.hot-update.json

      • 这个 js 脚本下载完后，会自动调用函数，传了三个函数：chunkId，新的模块内容，一个运行时函数，记录本次的 hash，下次热更就可以拿到。

      • apply 的关键函数 applyHandler，大概逻辑就是返回 dispose 和最终 apply 逻辑，最终的 apply 就是删除模块缓存，进行模块替换，替换完后，执行 accept 定义的回调，如下，下一次 require 或者 import 拿到的内容就是更新后的
      module.hot?.accept(['./print.js'], render)

      • apply 里面还有一个很关键的逻辑，寻找更新边界 getAffectedModuleEffects ,如果发现该模块或者父级都没有 module.hot?.accept，那么会抛出一个错误，最后直接 reload
      • 寻找更新边界的关键点在于维护模块的父子关系，这个关系是在模块 require 其他模块的时候维护的，webpack 重写了 require 逻辑。通过父子关系，就可以一直往上冒泡查找是否有父级模块接受了更新，并且可以拿到回调函数。
