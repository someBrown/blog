    # CJS 和 ESM

    这一部分重点会讲CJS和ESM是怎么处理循环导入的。

    # 差异

    这两个模块体系，就好像“阿金”和“阿锦”的差异。

    先来看看CJS和ESM有什么不同，这里直接引用阮一峰 (opens new window)老师的《ECMAScript 6 入门》 (opens new window)对两者三个重大差异的总结

    CommonJS 模块输出的是一个值的拷贝，ES6 模块输出的是值的引用。 CommonJS 模块是运行时加载，ES6 模块是编译时输出接口。 CommonJS 模块的 require()是同步加载模块，ES6 模块的 import 命令是异步加载，有一个独立的模块依赖的解析阶段。

    # 怎么理解上面关于 CJS 的特性

    首先对于 CJS 模块是运行时加载我完全认同，并且加载的是一个module.exports这个对象，这个对象只有在脚本运行完才会生成。但是对于“输出的是一个值的拷贝”我认为是不准确的，应该是如果输出的是基本类型，才是值拷贝，而如果输出的是非基本类型，比如对象，那输出的其实是这个对象的引用。下面看一个例子 🌰

    _load方法会稍微复杂点，主要功能是解析我们传入的路径，然后根据路径判断是否有缓存，这里的缓存非常重要，和我们循环导入的主题密切相关。如果缓存里面没有，那么就开始加载这个模块并且缓存下来，最后返回我们非常熟悉的module.exports对象

    MyModule._load = function (request) {
  // request是传入的路径
  const filename = MyModule._resolveFilename(request)

  // 先检查缓存，如果缓存存在且已经加载，直接返回缓存
  const cachedModule = MyModule._cache[filename]
  if (cachedModule) {
    return cachedModule.exports
  }

  // 如果缓存不存在，我们就加载这个模块
  const module = new MyModule(filename)

  // load之前就将这个模块缓存下来
  MyModule._cache[filename] = module

  // 开始加载文件
  module.load(filename)

  return module.exports
}

    _compile简单实现如下，MyModule.wrap就是拼接字符串，使得我们平常书写的模块成为一个完整的function表达式，但是这里得到还是一个字符串，所有又通过vm.runInThisContext处理得到一个可执行的function对象，和浏览器中也可以通过字符串构造出function对象类似，最后就是执行这个方法了

    MyModule.prototype._compile = function (content, filename) {
  var self = this
  // 获取包装后函数体
  const wrapper = MyModule.wrap(content)

  // vm是nodejs的虚拟机沙盒模块，runInThisContext方法可以接受一个字符串并将它转化为一个函数
  // 返回值就是转化后的函数，所以compiledWrapper是一个函数
  const compiledWrapper = vm.runInThisContext(wrapper, {
    filename,
  })
  const dirname = path.dirname(filename)

  const args = [self.exports, self.require, self, filename, dirname]
  return compiledWrapper.apply(self.exports, args)
}

    # 怎么理解上面关于 ESM 的特性

    ES6 模块是编译时输出接口，输出的是值的引用

    首先注意 ⚠️ 上面CJS的require和module.exports，都是普通的对象，而与此相对应的  **ESM 中的  import  和  export  则是关键字，是 ECMAScript 标准的一部分。**关键字和对象有一个非常大的区别，关键字我们通过代码无法模拟，比如你无法定义一个名为import的对象。

    “ES6 模块是编译时输出接口”这句话真的令人一头雾水，什么是接口？可以知道的是这个接口肯定不是平时服务端提供给我的那种接口。下面我用自己的理解翻译一下。

    先简单回顾一下JS的执行过程，尽管通常认为JS是“解释型”语言，但是JS代码在执行前，其实还是有编译的过程。举个例子

    console.log(name)
let name = '阿锦'
//var name = '阿锦'
    1
    2
    3

    上面这段代码在，我们习惯看作一个声明，但实际上 JS 引擎会当成let name和name = ‘阿锦’两个声明，第一个发生在编译阶段，第二个赋值发生在执行阶段。编译阶段把name放到“暂存性死区”，在执行时访问“暂存性死区”的变量会抛出异常，只有执行过变量声明语句后，才会从“暂存性死区”中移出。如果是var声明，那就是直接在当前作用域声明一个变量并命名为name,值是undefined。

    所以ES6 模块是编译时输出接口这句话可以理解为，ES6的模块解析发生在编译阶段，import   模块中引入的值指向了  export中导出的值，并且这个值import后是只读的。

    # CJS 的循环引用

    “循环加载”（circular dependency）指的是，a脚本的执行依赖b脚本，而b脚本的执行又依赖a 脚本。CJS 模块因为缓存机制，出现循环加载并不会出现“死锁”。下面看一下这个例子的执行过程：

    //index.js
require('./a.js')

//a.js
exports.done = false
var b = require('./b.js')
console.log('在 a.js 之中，b.done = %j', b.done)
exports.done = true
exports.value = 'a的值'
console.log('a.js 执行完毕')

//b.js
exports.done = false
var a = require('./a.js')
console.log('在 b.js 之中，a的值', a.value)
console.log('在 b.js 之中，a.done = %j', a.done)
exports.done = true
console.log('b.js 执行完毕')

    1. 首先 index.js 引入 a.js，注意这里，根据上面我们实现的简易模块加载机制，这里 a.js 会被加入到缓存，a 模块的exports是一个空对象。
    2. a.js 执行，exports对象新增了一个属性done，接着引入 b.js
    3. b.js 执行，同上，exports对象新增了一个属性done，然后引入了 a.js，这时候引入 a.js，因为缓存的存在，会直接返回 a.js 的 exports 对象{done:false}。所以 b.js 继续执行，拿到的a.value是undefined，因为 a.js 还没执行到exports.value = 'a的值' 赋值这一行。
    4. 最后 b.js 执行完后，a.js 继续执行直到结束。

    所以 CJS 一旦出现某个模块被"循环加载"，就只输出已经执行的部分，还未执行的部分不会输出。

    js 文件是一个IIFE，我们先来看看函数体，和我们上面实现的模块加载器非常相似。首先是定义了一个installedModules的缓存对象，接着是定义require 方法，下面是webpack一些辅助函数，我们可以忽略，最后主动调用 __webpack_require__((__webpack_require__.s = 1))，传入的moduleId的是 1。可以理解为这里是开始加载第一个模块，是这个 js 文件运行的真正起点。

    // webpackBootstrap
function noName(modules) {
  const installedModules = {}

  // The require function
  function __webpack_require__(moduleId) {
    // Check if module is in cache
    if (installedModules[moduleId]) {
      return installedModules[moduleId].exports
    }
    // Create a new module (and put it into the cache)
    const module = (installedModules[moduleId] = {
      i: moduleId,
      l: false,
      exports: {},
    })

    // Execute the module function
    modules[moduleId].call(
      module.exports,
      module,
      module.exports,
      __webpack_require__
    )

    // Flag the module as loaded
    module.l = true

    // Return the exports of the module
    return module.exports
  }

  // expose the modules object (__webpack_modules__)
  __webpack_require__.m = modules

  // expose the module cache
  __webpack_require__.c = installedModules

  // identity function for calling harmony imports with the correct context
  __webpack_require__.i = function (value) {
    return value
  }

  // define getter function for harmony exports
  __webpack_require__.d = function (exports, name, getter) {
    if (!__webpack_require__.o(exports, name)) {
      Object.defineProperty(exports, name, {
        configurable: false,
        enumerable: true,
        get: getter,
      })
    }
  }

  // getDefaultExport function for compatibility with non-harmony modules
  __webpack_require__.n = function (module) {
    const getter =
      module && module.__esModule
        ? function getDefault() {
            return module.default
          }
        : function getModuleExports() {
            return module
          }
    __webpack_require__.d(getter, 'a', getter)
    return getter
  }

  // Object.prototype.hasOwnProperty.call
  __webpack_require__.o = function (object, property) {
    return Object.prototype.hasOwnProperty.call(object, property)
  }

  // __webpack_public_path__
  __webpack_require__.p = '/'

  // Load entry module and return exports
  return __webpack_require__((__webpack_require__.s = 1))
